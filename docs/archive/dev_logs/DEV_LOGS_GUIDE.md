# 개발 로그 작성 가이드 (최신)

## 0. 적용 범위

- 대상: `docs/dev_logs/**` 아래 모든 개발 로그 문서
- 목적: 작업 맥락, 변경 근거, 검증 결과를 팀이 재현 가능하게 기록

## 1. 필수 원칙

1. `dev_logs` 작성 전 이 문서를 반드시 먼저 확인합니다.
2. 코드 변경이 있는 작업은 같은 브랜치/PR 흐름에서 개발 로그를 함께 남깁니다.
3. `작업자`는 사람 이름으로 작성합니다.
4. `CloudDX`, `Codex`, `AI`, `Assistant` 같은 시스템/도구 이름을 작업자로 쓰지 않습니다.
5. UI 변경이 있으면 스크린샷을 첨부합니다.
6. 모호한 표현(`fix`, `수정함`)만 남기지 말고 변경 범위와 결과를 구체적으로 적습니다.

## 2. 작성 시점

1. 기능 구현 또는 버그 수정 완료
2. 로컬/서버 검증 완료
3. 개발 로그 작성
4. 커밋/푸시

## 3. 파일명/위치 규칙

- 파일명: `YYYY-MM-DD_작업주제.md`
- 예시: `2026-02-25_gitlab_cicd_notification_setup.md`
- 위치: `docs/dev_logs/<주차 폴더>/`
- 스크린샷: `docs/dev_logs/screenshots/YYYY-MM-DD/`
- 인코딩: `UTF-8`로 저장 (깨짐 방지)

## 4. 필수 섹션(누락 금지)

1. 작업 요약
2. 상세 변경 사항
3. 작업 중 발생 이슈 및 대응
4. 결과(검증 포함)
5. 커밋 로그
6. 후속 작업/리스크

## 5. 작성 규칙

1. `작업 일시`는 `YYYY-MM-DD` 또는 필요한 경우 시간까지 명시합니다.
2. `작업자`는 실명 기준으로 작성합니다.
3. 변경 파일/리소스는 가능한 한 경로를 포함합니다.
4. 검증 결과는 명령/엔드포인트/건수처럼 재현 가능한 형태로 작성합니다.
5. 스크린샷은 문서에서 상대 경로로 링크합니다.
6. 커밋 로그 구간은 작업 날짜와 일치시킵니다.

## 6. GitLab 기준 권장 항목

- 브랜치명 (`develop`, `feature/*`)
- 파이프라인 결과(성공/실패, 필요 시 링크)
- 배포/롤아웃 확인 결과

## 8. UI 스크린샷 규칙

1. 저장 경로: `docs/dev_logs/screenshots/YYYY-MM-DD/`
2. 파일명은 화면 의미가 드러나게 작성합니다 (`login_page.png`, `portfolio_chart.png`)
3. 문서에는 상대 경로로 삽입합니다.

예시:

```markdown
## UI 스크린샷
![로그인 페이지](screenshots/2026-02-25/login_page.png)
```
