# 🔧 Troubleshooting Report (2026-02-13 ~ 2026-02-18)

## 이슈 1: Frontend WebSocket 연결 오류

### 📝 증상

- **환경**: 로컬 개발 환경 (Mac)
- **현상**: 브라우저 콘솔에 `ws://localhost:3000/api/proxy/api/v1/market/ws` 연결 실패 에러 반복 출력
- **에러 메시지**: `WebSocket connection to 'ws://localhost:3000/api/proxy/api/v1/market/ws' failed`

### 🔍 원인 분석

- Next.js API Route를 통한 WebSocket 프록시 설정 미흡
- 로컬 환경에서 백엔드 WebSocket 엔드포인트 직접 연결 필요

### ✅ 해결 방법 (임시)

- 로컬 개발 시 `.env.local`에서 `NEXT_PUBLIC_WS_URL`을 백엔드 직접 연결로 설정
- K8s 환경에서는 Nginx Ingress의 WebSocket 어노테이션으로 해결 예정:
  ```yaml
  nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
  nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"
  nginx.ingress.kubernetes.io/websocket-services: "backend-svc"
  ```

### 📚 Lessons Learned

- WebSocket은 HTTP 업그레이드 핸드셰이크가 필요하므로 Ingress 레벨에서 별도 설정 필수
- Next.js API Route는 WebSocket을 기본 지원하지 않음 (별도 서버 필요)

## 이슈 2: Backend Redis 연결 오류 (로컬)

### 📝 증상

- **환경**: 로컬 개발 환경 (Mac)
- **에러 메시지**: `Error 61 connecting to localhost:6379. Connection refused.`
- **현상**: 백엔드 서버 시작 후 Redis 관련 기능 사용 시 오류

### 🔍 원인 분석

- 로컬 Mac에 Redis가 설치/실행되지 않은 상태
- 백엔드 `.env`의 `REDIS_URL=redis://localhost:6379` 설정이 로컬에서 연결 불가

### ✅ 해결 방법

- 로컬 개발 시 Docker로 Redis 실행:
  ```bash
  docker run -d -p 6379:6379 redis:alpine
  ```
- 또는 `REDIS_URL`을 VirtualBox VM의 Redis로 SSH 터널링:
  ```bash
  ssh -L 6379:localhost:6379 user@192.168.56.XX -N
  ```

### 📚 Lessons Learned

- 로컬 개발 환경 체크리스트에 Redis 실행 여부 추가 필요
- K8s 환경에서는 `redis-svc.tutum-data:6379`로 자동 해결됨

## 이슈 3: Kakao OAuth KAKAO_CLIENT_ID 누락

### 📝 증상

- **환경**: 로컬 개발 환경
- **현상**: 카카오 로그인 버튼 클릭 시 OAuth 리다이렉트 실패
- **에러**: `KAKAO_CLIENT_ID is not set`

### 🔍 원인 분석

- `.env` 파일에 `KAKAO_CLIENT_ID` 키가 없거나 빈 값
- 카카오 개발자 콘솔에서 앱 키 발급 필요

### ✅ 해결 방법 (임시)

- 카카오 로그인 기능 로컬 테스트 시 스킵
- K8s 배포 시 `backend-secret`에 `KAKAO_CLIENT_ID` 추가 예정

### 📚 Lessons Learned

- OAuth 키는 환경별로 별도 관리 필요 (로컬/스테이징/프로덕션)
- K8s Secret에 모든 OAuth 키 통합 관리

## 이슈 4: Python 가상환경 의존성 충돌

### 📝 증상

- **환경**: 로컬 Mac
- **현상**: `pip install -r requirements.txt` 실행 시 패키지 버전 충돌
- **에러**: `ERROR: pip's dependency resolver does not currently take into account all the packages that are installed`

### 🔍 원인 분석

- 기존 가상환경에 다른 버전의 패키지가 설치되어 있어 충돌
- `framer-motion` 등 일부 패키지 누락

### ✅ 해결 방법

```bash
# 기존 venv 삭제 후 재생성
rm -rf venv
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

### 📚 Lessons Learned

- 의존성 충돌 시 가상환경 재생성이 가장 빠른 해결책
- `requirements.txt`에 버전 고정 (`==`) 권장
