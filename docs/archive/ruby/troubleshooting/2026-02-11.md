# Troubleshooting Log (2026-02-11)

## Summary

### 1. MinIO Cluster Communication issue

- **What broke**: Backend (Node1) could not reach MinIO (Node2) via the bridged public IP `192.168.0.28:9000`.
- **Impact**: Storage service initialization failed on backend startup.
- **Root cause**: Host system's port forwarding and VM network isolation prevented direct bridged communication on specific ports between VMs.
- **Fix**: Used the internal VirtualBox host-only network IP (`192.168.56.12`) which provides clear direct communication between cluster nodes.
- **Prevention**: Always prefer internal subnet IPs for inter-cluster traffic.

### 2. Email Worker Module Path Error

- **What broke**: `python workers/email_worker.py` failed with `ModuleNotFoundError: No module named 'app'`.
- **Impact**: Background email processing was unavailable.
- **Root cause**: The worker script couldn't find the `app` package because the parent directory wasn't in `PYTHONPATH`.
- **Fix**: Added dynamic path insertion at the beginning of the worker script: `sys.path.insert(0, os.path.abspath(os.path.join(os.path.dirname(__file__), '..')))`.
- **Prevention**: Use standard entry point patterns or proper package installation for background workers.

## Fix (Step-by-step)

### Internal IP Routing

1. Identify internal IPs using `hostname -I`.
2. Test connectivity between nodes using the identified IPs (`56.x` subnet).
3. Update `.env` `MINIO_ENDPOINT` to the internal IP.

## Lessons Learned

- **Internal First**: For VM-to-VM communication in complex setups, local bridged IPs (forwarded by host) are less reliable than dedicated internal network interfaces.
- **Path Isolation**: Workers in subdirectories require careful management of the Python search path if they depend on the main application code.
