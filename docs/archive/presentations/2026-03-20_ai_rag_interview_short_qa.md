# TUTUM AI/RAG 짧은 면접 답변 정리

## 1. 문서 목적
- 발표 후 받을 수 있는 AI/RAG 관련 질문에 대해, 너무 어렵지 않게 바로 말할 수 있도록 정리한 문서
- 실제 면접에서 답하는 말투에 가깝게 짧게 정리

## 2. 한 줄 설명
- 저희 AI는 질문을 바로 LLM에 보내는 게 아니라, 포트폴리오, 실시간 시세, 뉴스 데이터를 먼저 모아서 같이 전달하는 RAG 구조입니다.

## 3. 예상 질문과 답변

### Q1. 왜 RAG를 썼나요?
- 저희 서비스는 투자나 자산 관련 질문이 많아서, 그냥 LLM만 쓰면 최신 뉴스나 사용자 자산 상황이 반영되지 않는 문제가 있었습니다.
- 그래서 포트폴리오, 실시간 시세, 뉴스 데이터를 먼저 조회한 뒤 같이 넣어주는 RAG 구조를 사용했습니다.

### Q2. AI가 어떤 데이터를 보고 답하나요?
- 크게 세 가지입니다. 사용자 포트폴리오, 실시간 시세, 뉴스 데이터입니다.
- 포트폴리오는 MariaDB, 시세는 Upbit와 KIS, 뉴스는 Elasticsearch와 MongoDB에서 가져오도록 구성했습니다.

### Q3. BM25와 kNN을 왜 같이 썼나요?
- BM25는 종목명이나 티커처럼 정확한 단어를 찾는 데 강하고, kNN은 의미가 비슷한 문서를 찾는 데 강합니다.
- 금융 질문은 둘 다 중요해서 두 방식을 같이 쓰는 하이브리드 검색을 적용했습니다.

### Q4. MongoDB와 Elasticsearch를 둘 다 쓴 이유가 뭔가요?
- MongoDB는 뉴스 원문을 저장하는 용도이고, Elasticsearch는 검색을 빠르게 하기 위한 용도입니다.
- 쉽게 말하면 MongoDB는 원본 저장소, Elasticsearch는 검색용 엔진이라고 보시면 됩니다.

### Q5. Elasticsearch가 안 되면 AI도 같이 멈추나요?
- 완전히 멈추지는 않도록 fallback을 넣었습니다.
- 먼저 Elasticsearch를 쓰고, 안 되면 MongoDB에 저장된 원문 데이터를 대신 참고하도록 만들었습니다.

### Q6. 뉴스는 얼마나 최신으로 반영되나요?
- 뉴스는 질문 시점에 바로 외부 API를 치는 방식이 아니라, 수집기가 Kafka로 넣고 consumer가 적재하는 구조입니다.
- 그래서 내부에 적재된 최신 뉴스 데이터를 기준으로 답하도록 만들었습니다.

### Q7. 외부 API rate limit이나 실패는 어떻게 대응했나요?
- 시세 쪽은 Redis 캐시와 `last_good` 값을 두어서, 외부 API가 실패해도 직전 정상값을 보여주도록 했습니다.
- 그래서 사용자 화면에 0원이나 비정상 값이 뜨지 않도록 보완했습니다.

### Q8. AI 답변이 느릴 때는 어디부터 보나요?
- 포트폴리오 조회, 시세 조회, 뉴스 검색, LLM 응답 구간으로 나눠서 봅니다.
- 실제로는 Elasticsearch 연결 주소가 잘못돼 뉴스 검색 단계가 느려진 적이 있었고, 그 부분을 수정해서 개선한 경험이 있습니다.

### Q9. AI 답변 품질은 어떻게 개선했나요?
- 모델만 바꾸기보다 검색 문맥이 맞는지를 먼저 봤습니다.
- 예를 들어 특정 종목 질문인데 포트폴리오 편향 때문에 다른 뉴스가 섞이는 문제가 있어서, 질문 의도 판별과 검색 키워드 조합 로직을 수정했습니다.

### Q10. 왜 Bedrock과 Titan을 썼나요?
- 현재 인프라가 AWS 중심이라서 운영과 권한 관리 측면에서 일관성이 좋았습니다.
- 같은 클라우드 안에서 LLM과 임베딩을 같이 쓰는 게 운영상 더 편했습니다.

### Q11. 현재 사용 중인 모델은 무엇인가요?
- 현재는 Amazon Bedrock의 Claude Sonnet 4.6을 사용하고 있습니다.
- 임베딩은 Titan Embed v2를 사용합니다.

### Q12. 포트폴리오 분석 요청이 오면 어떻게 동작하나요?
- 먼저 사용자 포트폴리오를 조회하고, 관련 시세와 뉴스 문맥을 같이 모읍니다.
- 그다음 이 데이터를 Bedrock Claude에 전달해서 답변을 만들고, 결과는 스트리밍 형태로 프론트에 보내는 구조입니다.

## 4. 마지막 정리 멘트
- 저희 AI의 핵심은 단순히 모델을 붙인 게 아니라, 사용자 자산과 최신 데이터를 먼저 모아서 근거 기반으로 답하도록 만든 운영형 RAG 구조라는 점입니다.
